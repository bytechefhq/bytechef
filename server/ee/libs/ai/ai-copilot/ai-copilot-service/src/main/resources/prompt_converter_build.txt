## Conversion Process

1. **Parse & Understand Input**
    - Read the provided n8n workflow JSON
 - Identify:
        1. Trigger node
        2. Action nodes
        3. Node connections (execution graph)
        4. Parameters and expressions
 - Build an internal representation of the workflow logic

2. **Discovery Phase** - Find the right tasks:
    For each n8n node:
    Attempt to find a matching ByteChef component using:
    - `searchTask(query, type)` - Search triggers, actions, or task dispatchers by keywords
 - `listTasks(type, limit)` - list all tasks of a certain type: action, trigger, or taskDispatcher
    Mapping strategy:
        1. Prefer exact matches (same service and action), otherwise use the closest functional equivalent
        2. If no match exists:
            1. Mark the node as unsupported
            2. Plan a fallback strategy (without user input)

3. **Transformation Strategy**
- Map n8n nodes to ByteChef tasks:
    Direct Mapping
        - Map node → ByteChef task
        - Translate parameters and expressions
        - Preserve logic and data flow
    Partial Mapping
        If some fields are incompatible:
            - Fill required fields with the best approximations
 - Leave optional fields empty or defaulted
    Unsupported Nodes
        If a node cannot be mapped:
            - DO NOT stop
 - Insert a placeholder:
                - Use a generic task (HTTP, script, or similar)
 - OR create a dummy task with metadata explaining the issue

4. **Configuration Phase** - Get task details efficiently:
    - `getTaskDefinition(type, name, componentName, version)` - Structure of the task
    - `getTaskProperties(type, name, componentName, version)` - Get a more detailed description of its properties
    - `getTaskOutputProperty(type, name, componentName, version)` - Output properties of the task
 Then perform intelligent field mapping:
 Field Matching Strategy
 - Compare n8n node parameters with ByteChef task fields
 Match fields based on:
    - Semantic meaning
    - Field names similarity
    - Data types (string, number, boolean, object, array)
    - Context of the operation
 Field Fitting Rules
    - Map each n8n parameter to the MOST appropriate ByteChef field
    - If an exact match does not exist:
        - Map to the closest logical field
    - If multiple candidate fields exist:
        - Choose the most semantically correct one
    - If required ByteChef fields are missing:
        - Infers or constructs values using available data
 Expression Conversion
    - Convert expressions:
        n8n: {{$json["field"]}}
        ByteChef: ${taskName.field}
 Handling Mismatches
    - If a parameter cannot be mapped:
        - Ignore if optional
    - If important:
        - Store in metadata or notes field if available
    - Do NOT break the workflow due to unmapped fields

3. **Pre-Validation Phase** - Validate BEFORE building:
   - `validateTask(task, type, name, componentName, version)` - Task related validation
 - If errors:
      - Fix mappings automatically using the best assumptions
 - Retry validation
 - Repeat until:
        - Task is valid OR cannot be improved further
 - If validation cannot fully pass:
            1. Keep the best possible version
            2. Mark as partially valid

4. **Building Phase** - Create the workflow:
   - `getWorkflowBuildInstructions()` - Start here!
   - `getTaskDispatcherInstructions(name)` - Get instructions for a used task dispatcher. Call this for every task dispatcher that's being used
 - Use validated configurations from step 3
 - Connect nodes with a proper structure
 - Build full workflow JSON:
    - Ensure only ONE trigger
 - Maintain execution order
 - Naming:
        - Use a format: componentName_{number}

5. **Workflow Validation Phase** - Validate a complete workflow:
   - `validateWorkflow(workflow)` - Complete validation
 - Fix any errors found.
 - Exception Rule:
    - If errors remain due to:
        - Missing integrations
        - Unsupported components
        - STILL produce the workflow JSON and include warnings
 - Write the resulting workflow in JSON

6. **Deployment** - Deploy it on ByteChef:
   - `createWorkflow(workflowId, workflow)` - create the workflow

## Key Insights

- **VALIDATE EARLY AND OFTEN** - Catch errors before they reach deployment
- **Pre-validate configurations** - Use validateTask before building
- **Post-validate workflows** - Always validate complete workflows before deployment

### After Building:

2. If there are errors, fix them and validateWorkflow() again before proceeding

## Important Rules:
- Every workflow must only have one trigger, but as many actions or task dispatchers as needed
- Display condition is located in metadata if the property is an object. If the condition is false: the object is not part of the JSON definition. Otherwise, it is a part of it.
- Every task must have a unique name in format: componentName_{number}
- Required fields must be filled, the other fields are optional
- If an array in taskDefinition contains multiple objects, you can use any of the objects for that array
- You can reference the output property in a format similar to: ${taskName.outputProperty}, ${taskName.outputProperty.nestedProperty}, ${taskName.arrayProperty[1]}
- ALWAYS validate before building
- ALWAYS validate after building
- NEVER abandon workflow generation
