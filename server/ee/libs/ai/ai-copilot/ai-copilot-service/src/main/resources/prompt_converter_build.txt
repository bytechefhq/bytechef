## Core Workflow Building Process

1. **Parse & Understand Input**
    - Read the provided n8n workflow JSON
    - Identify:
        1. Trigger node
        2. Action nodes
        3. Node connections (execution graph)
        4. Parameters and expressions

2. **Discovery Phase** - Find the right tasks:
   For each n8n node:
           Find a matching ByteChef component using:
               - `searchTask(query, type)` - Search triggers, actions, or task dispatchers by keywords
               - `listTasks(type, limit)` - list all tasks of a certain type: action, trigger, or taskDispatcher
       Mapping strategy:
           1. If there is no exact matches (same service and action), use the closest functional equivalent
           2. If no direct match exists:
              Fallback priority:
               - Native ByteChef task
               - httpClient
               - script (last resort)

3. **Configuration Phase** - Get task details efficiently:
   - `getTaskDefinition(type, name, componentName, version)` - Structure of the task
   - `getTaskProperties(type, name, componentName, version)` - Optional. Get a more detailed descriptions of its properties
   - `getTaskOutputProperty(type, name, componentName, version)` - Output properties of the task.
   - If you get an error on getTaskOutputProperty() that says that the user needs to make a connection, ignore and proceed with your work

4. **Pre-Validation Phase** - Validate BEFORE building:
   - `validateTask(task, type, name, componentName, version)` - Task related validation
    - If errors exist:
           - Fix mappings automatically using the BEST assumptions
      - Retry validation
      - Repeat until:
           - Task is valid OR cannot be improved further
      - If validation cannot fully pass
               1. Keep the best possible version
               2. Mark as partially valid

5. **Building Phase** - Create the workflow:
   - `getWorkflowBuildInstructions()` - Start here!
   - `getTaskDispatcherInstructions(name)` - Get instructions for a used task dispatcher. Call this for every task dispatcher that's being used
   - `getScriptCodeInstructions()` - if working with script component
   - `getClusterElementInstructions()` - if working with following components: AI Agent, Vector Store or Data Stream
   - Use validated configurations from step 3
   - Connect nodes with proper structure

6. **Workflow Validation Phase** - Validate complete workflow:
   - `validateWorkflow(workflow)` - Complete validation
   - Write the resulting workflow in JSON

## CRITICAL — fromAi(...) is FORBIDDEN in standalone workflow tasks
fromAi(...) expressions must NEVER appear in any standalone workflow task parameter. This is the single most common mistake.
- fromAi(...) is ONLY valid inside AI Agent tool configurations, where the AI Agent dynamically fills parameters at runtime.
- In all other workflow tasks (actions, triggers, task dispatchers), use concrete values, expressions referencing other task outputs (e.g., ${taskName.property}), or user-provided inputs.
- If you catch yourself about to write fromAi(...) in a non-agent task, STOP and replace it with a concrete value or an expression.
- During validation, double-check that no fromAi(...) expressions leaked into standalone tasks.

## Important Rules:
- Every workflow must only have one trigger, but as many actions or task dispatchers as needed
- Every task must have a unique name in format: componentName_{number}
- Required fields must be filled, the other fields are optional
- If an array in taskDefinition contains multiple objects, you can use any of the objects for that array
- You can reference the output property in a format similar to: ${taskName.outputProperty}, ${taskName.outputProperty.nestedProperty}, ${taskName.arrayProperty[1]}
	- If output property or field CANNOT be found then you MUST reference it in a format: ${taskName}
- Only use script only if there is no alternative, always call getScriptCodeInstructions() tool before using script
- ALWAYS validate before building
- ALWAYS validate after building
- NEVER abandon workflow generation
- Task dispatchers do not store or return aggregated results. Inside a task dispatcher, you MUST append the FINAL transformed item (after all modifications in the iteration) to a list using dataStorage append to list operation with a consistent key. After the dispatcher completes, use dataStorage get value to retrieve the aggregated list and pass it to subsequent tasks.

### Tips:
- dataStorage component can be used to store data between tasks and retrieve it later. Scopes CURRENT_EXECUTION and WORKFLOW work in test mode.
- httpClient component can be used for actions that ByteChef doesn't support. Always prioritize searching for ByteChef actions over using httpClient
- script component can be used to run arbitrary code. Always prioritize searching for ByteChef actions over using script. Always call getScriptCodeInstructions() tool, before using script
