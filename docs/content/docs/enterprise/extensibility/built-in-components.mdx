## Anatomy of a built-in component

Each component is a self-contained module under `server/libs/modules/components/`:

```
my-component/
├── build.gradle.kts
├── src/main/java/             # Component handler, actions, triggers
├── src/main/resources/
│   ├── README.md              # Component documentation
│   └── openapi.yaml           # When wrapping a REST API
└── src/test/java/             # Unit and integration tests
```

The same scaffold applies to [custom components](/enterprise/extensibility/custom-components) — built-in vs. custom is a distribution distinction, not an architectural one.

## Versioning

Components are versioned (`v1`, `v2`, ...). When a third-party API breaks compatibility, a new component version is cut so existing workflows can stay on the old one until they migrate. The component reference docs always show available versions.

## Auto-generated docs

Every component ships a `README.md` describing its actions, triggers, and connection. These are aggregated into the [Component Reference](/reference) section of the docs and regenerated with each build:

```
./gradlew generateDocumentation
```

## When the catalog doesn't cover your need

In rough order of effort:

1. **Polyglot script step** — paste a few lines of JS/Python and call the API directly. ([Polyglot scripting](/enterprise/extensibility/polyglot-scripting))
2. **HTTP Client component** — typed HTTP request without any new component code.
3. **API connector** — point at an OpenAPI spec, get a generated component. ([API connector](/enterprise/extensibility/custom-components#auto-generated-api-connectors-coming-soon) *(coming soon)*)
4. **Custom component** — fully hand-written when none of the above fit. ([Custom components](/enterprise/extensibility/custom-components))

## Source-available, modifiable

Because every built-in component is in the open-source repo, when one of them doesn't quite do what you need, you can:

- Fork the component module into your own repo.
- Modify it to your taste.
- Contribute the change back via PR.

You're never blocked waiting for a vendor to ship a feature.
