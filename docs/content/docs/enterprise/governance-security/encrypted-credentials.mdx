ByteChef workflows talk to external systems on behalf of your users — Salesforce, Stripe, GitHub, internal APIs. Those credentials are some of the most sensitive material in your stack. Enterprise's credential storage is built around two guarantees:

1. **No plaintext at rest.** Credentials are encrypted before they hit the database; the unencrypted form only exists in memory at execution time.
2. **No leakage in observability.** Logs, traces, and execution outputs are scrubbed for credential values; you cannot accidentally surface a token by enabling DEBUG logging.

## How the encryption works

Each credential is encrypted with a workspace-scoped key. The workspace key is itself wrapped by a master key supplied at boot time via environment variable (or, when you wire one in, from your [external secret manager](#external-secret-managers-coming-soon) *(coming soon)*).

```
Plaintext credential
    ↓ AES-GCM with workspace key
Encrypted credential (stored in Postgres)
    ↓ (workspace key is wrapped by master key)
Master key (BYTECHEF_ENCRYPTION_KEY or external vault)
```

This layered model means:

- A leaked database backup is useless without the master key.
- Rotating a workspace key only requires re-wrapping its credentials, not re-encrypting them.
- Different workspaces (or different tenants in an [embedded](/embedded/white-label-execution) deployment) cannot decrypt each other's secrets even if they share the same database.

## What is treated as a credential

- **OAuth tokens** (access and refresh).
- **API keys** and bearer tokens.
- **Basic auth** username/password pairs.
- **Certificate material** and private keys when used for client auth.
- **Custom auth fields** declared by component definitions.

If a connection definition marks a property as sensitive, the platform treats it as a credential end-to-end — encryption at rest, masking in UI, scrubbing in logs.

## Scrubbing across the stack

The same set of "this is sensitive" markers drives scrubbing in three places:

- **Logs.** Sensitive values are replaced with `***` before any log line leaves the JVM.
- **Traces.** OpenTelemetry spans omit the sensitive properties; only their names are recorded.
- **Execution history.** Workflow execution outputs persist the masked form, not the secret.

You don't need to remember which fields are sensitive on which component — the component author declares it once, and the platform enforces it everywhere.

## Rotation

Rotating a credential happens in one place — the connection edit screen. Because workflows reference connections by ID (not by inlining the secret), every running and future execution picks up the new value without redeploying any workflow.

## External secret managers (coming soon)

Encrypted credential storage is enough for most teams. External secret managers exist for teams that need stronger guarantees:

- **Centralized rotation.** Your security team already rotates secrets in your vault on a schedule. Mirroring those rotations into ByteChef is operational toil that breaks.
- **Provable separation of duties.** The team that runs ByteChef should not have read access to production secrets. Externalizing puts secrets behind a wall ByteChef admins can't cross.
- **Compliance regimes** that mandate secrets live in a FIPS-validated vault or an HSM-backed store.
- **Cross-tool consistency.** The same secret feeds your application, your CI/CD, and your workflows — one source of truth, one rotation procedure.

| Provider | Auth methods | Notes |
|---|---|---|
| **HashiCorp Vault** | Token, AppRole, Kubernetes auth | KV v2 secret engines. |
| **AWS Secrets Manager** | IAM role (recommended), access key | Native VPC endpoint support. |
| **Azure Key Vault** | Managed identity, service principal | Secret references via Azure SDK. |

**How injection works.** Instead of storing the secret value, the connection record stores a reference (e.g., `vault://kv/data/prod/stripe?field=api_key`, `awssm://prod/integrations/stripe?field=api_key`, `azurekv://prod-vault/stripe-api-key`). At execution time the platform resolves the reference against the configured vault, caches the value in memory for the duration of the execution, injects it into the action's sensitive properties, and drops the value when the execution completes. The secret never lands in ByteChef's database, never appears in logs, and never persists past the execution.

**Rotation propagation.** When you rotate a secret in your vault, the next workflow execution picks up the new value automatically — no ByteChef-side update needed. Long-running connections honor a configurable refresh interval so a rotation doesn't require a process restart.

**Mixed mode.** The two storage modes coexist — high-value production secrets in your vault, lower-stakes dev/test credentials in ByteChef's encrypted storage. Each connection chooses its own backend.

## Fine-grained connection visibility

Encryption protects the secret material; **visibility** controls who can even see the connection exists. A connection has exactly one scope at a time — `PRIVATE`, `WORKSPACE`, or `ORGANIZATION` — and widening it is an explicit, admin-gated, audited action rather than an out-of-band Slack DM.

See [Connection visibility](/enterprise/governance-security/connection-visibility) for the full model: the three scopes, who can promote and demote, the GraphQL mutations, and the audit events. Visibility composes with this page's encryption (it governs who can *use* a connection, not where its secret lives) and with [RBAC](/enterprise/governance-security/rbac) (which governs what they can *do* with it).

## Operational checklist

- [ ] Set `BYTECHEF_ENCRYPTION_KEY` to a 32-byte value generated by your KMS, not a hardcoded string.
- [ ] Back up the master key with the same rigor as your database backup.
- [ ] Plan for master-key rotation — it requires unwrapping and re-wrapping workspace keys (zero-downtime).
- [ ] Audit access to the connection edit screen via [RBAC](/enterprise/governance-security/rbac) — credential writes are a high-value privilege.
