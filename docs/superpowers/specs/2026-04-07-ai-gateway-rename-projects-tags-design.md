# AI Gateway: Rename, Projects & Dynamic Tags

**Date:** 2026-04-07
**Status:** Draft

## Overview

Three changes to the AI Gateway module:

1. **Rename** `AiLlmGateway` → `AiGateway` across all layers (Gradle modules, Java packages, classes, DB tables, GraphQL, REST, client)
2. **Projects** — new entity for grouping AI Gateway traffic with per-project setting overrides
3. **Dynamic Tags** — replace hardcoded tags with workspace-scoped DB-stored tag definitions

## 2. Projects

### Concept

An AI Gateway project is a **separate entity** from ByteChef automation projects. It exists within a workspace, identified by a unique slug, and groups AI Gateway traffic for independent configuration and spend tracking.

Inspired by [Merge Gateway Projects](https://docs.merge.dev/merge-gateway/features/projects).

### Domain Entity

```
AiGatewayProject
├── id (Long, PK)
├── workspaceId (Long, required)
├── name (String, required)
├── slug (String, required, unique per workspace)
├── description (String, optional)
├── routingPolicyId (Long, FK to AiGatewayRoutingPolicy, nullable)
├── compressionEnabled (Boolean, nullable)
├── retryMaxAttempts (Integer, nullable)
├── timeoutSeconds (Integer, nullable)
├── cachingEnabled (Boolean, nullable)
├── cacheTtlMinutes (Integer, nullable)
├── logRetentionDays (Integer, nullable)
├── createdBy (String)
├── createdDate (Instant)
├── lastModifiedBy (String)
├── lastModifiedDate (Instant)
├── version (Long)
```

Constraints:
- Unique: `(workspace_id, slug)`
- FK: `routing_policy_id` → `ai_gateway_routing_policy(id)` (nullable)

### Setting Resolution

All nullable project fields fall back to system defaults:

```
project.compressionEnabled    ?? true      (system default)
project.retryMaxAttempts      ?? 2
project.timeoutSeconds        ?? 30
project.cachingEnabled        ?? true
project.cacheTtlMinutes       ?? 60
project.logRetentionDays      ?? 30
project.routingPolicyId       ?? workspace default routing policy
```

Future: when `AiGatewayWorkspaceSettings` is added, the chain becomes `project → workspace → system default`. No changes needed to the project entity — only the resolution logic adds the workspace lookup.

### Request Flow

```
Request arrives with X-Project-Id header (slug)
  → Resolve project by slug + workspace
  → Resolve settings: project overrides ?? system defaults
  → Apply compression if enabled
  → Route via project's routing policy (or workspace default)
  → Check project budget (if exists), then workspace budget
  → Execute LLM call with resolved retry/timeout settings
  → Log request with projectId
  → Track spend under project
```

Requests **without** `X-Project-Id` use workspace defaults directly (current behavior, unchanged).

### DB Table

```sql
CREATE TABLE ai_gateway_project (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY NOT NULL,
    workspace_id BIGINT NOT NULL,
    name        VARCHAR(256) NOT NULL,
    slug        VARCHAR(256) NOT NULL,
    description VARCHAR(1024),
    routing_policy_id BIGINT,
    compression_enabled BOOLEAN,
    retry_max_attempts INTEGER,
    timeout_seconds INTEGER,
    caching_enabled BOOLEAN,
    cache_ttl_minutes INTEGER,
    log_retention_days INTEGER,
    created_by  VARCHAR(256) NOT NULL,
    created_date TIMESTAMP NOT NULL,
    last_modified_by VARCHAR(256) NOT NULL,
    last_modified_date TIMESTAMP NOT NULL,
    version     BIGINT,

    CONSTRAINT pk_ai_gateway_project PRIMARY KEY (id),
    CONSTRAINT uq_ai_gateway_project_workspace_slug UNIQUE (workspace_id, slug),
    CONSTRAINT fk_ai_gateway_project_routing_policy FOREIGN KEY (routing_policy_id) REFERENCES ai_gateway_routing_policy(id)
);

CREATE INDEX ix_ai_gateway_project_workspace ON ai_gateway_project(workspace_id);
```

### Budget Integration

The existing `AiGatewayBudget` table gains a nullable `project_id` column:

```sql
ALTER TABLE ai_gateway_budget ADD COLUMN project_id BIGINT;
ALTER TABLE ai_gateway_budget ADD CONSTRAINT fk_ai_gateway_budget_project FOREIGN KEY (project_id) REFERENCES ai_gateway_project(id);
```

Update unique constraint from `(workspace_id)` to `(workspace_id, project_id)` where `project_id` can be null (workspace-level budget has null project_id).

### Request Log & Spend Integration

Add `project_id` column to:
- `ai_gateway_request_log` — nullable, indexed
- `ai_gateway_spend_summary` — nullable

## 3. Dynamic Tags

### Concept

Replace the hardcoded tag list in `AiGatewayTagApiController` with workspace-scoped tag definitions stored in the database. Tags are documentation/validation hints for API consumers — they describe what metadata can be sent with requests.

Request-level tags remain freeform key-value pairs. The tag definitions help API consumers know what keys are available and which are required.

### Domain Entity

```
AiGatewayTag
├── id (Long, PK)
├── workspaceId (Long, required)
├── key (String, required)
├── description (String, optional)
├── required (Boolean, default false)
├── createdBy (String)
├── createdDate (Instant)
├── lastModifiedBy (String)
├── lastModifiedDate (Instant)
├── version (Long)
```

Constraints:
- Unique: `(workspace_id, key)`

### DB Table

```sql
CREATE TABLE ai_gateway_tag (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY NOT NULL,
    workspace_id BIGINT NOT NULL,
    key         VARCHAR(256) NOT NULL,
    description VARCHAR(1024),
    required    BOOLEAN NOT NULL DEFAULT FALSE,
    created_by  VARCHAR(256) NOT NULL,
    created_date TIMESTAMP NOT NULL,
    last_modified_by VARCHAR(256) NOT NULL,
    last_modified_date TIMESTAMP NOT NULL,
    version     BIGINT,

    CONSTRAINT pk_ai_gateway_tag PRIMARY KEY (id),
    CONSTRAINT uq_ai_gateway_tag_workspace_key UNIQUE (workspace_id, key)
);

CREATE INDEX ix_ai_gateway_tag_workspace ON ai_gateway_tag(workspace_id);
```

### REST API (Public)

The existing `TagApi` endpoint (`GET /api/ai-gateway/v1/tags`) returns tags from the database for the authenticated workspace instead of the hardcoded list.

### Spend Tracking by Tag

Request logs already store tags. The spend summary and dashboard can break down costs by any tag value. No schema change needed — this is a query-time concern.

## 4. Client Changes

### New Pages/Components

- **Projects page** — List, create, edit, delete AI Gateway projects. Accessible from AI Gateway navigation.
- **Project detail/edit form** — Name, slug, description, routing policy select, compression toggle, retry/timeout/caching/retention overrides.
- **Tags management** — List, create, edit, delete tag definitions in the Settings or a dedicated Tags page.

### Updated Components

- **AiGatewayDashboard** — Add project filter to monitoring charts and request log.
- **AiGatewayBudget** — Add project selector for per-project budgets.
- **Navigation** — Add "Projects" and "Tags" tabs to the AI Gateway page.

## 5. Files Not Changed

- **Settings page** — Stays display-only with hardcoded defaults. Future task adds `AiGatewayWorkspaceSettings` entity to make it DB-backed.
- **Compression implementation** — `AiGatewayContextCompressor` logic unchanged. Only the facade reads `compressionEnabled` from the resolved project (or falls back to `true`).
